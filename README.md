# Easy_Ev_Charging

这是一个用原生PHP和MariaDB写的充电站管理与充电订单系统。

这个项目最早来自我大学期间做过的一个课程作业。原来的版本基本是传统的多页面PHP脚本，页面、SQL 和业务逻辑等并没有可以梳理。后来准备求职项目时，我决定以该项目为基础进行重构，将数据库和代码进行了重新设计。

现在的版本包括普通用户端和管理员端，除了基本的登录、站点查询和后台管理外，我主要花时间处理了充电订单的事务与并发、状态联动、计费、评价资格、安全、日志和自动化测试。

项目仍然是一个个人项目，不是准备直接用于真实充电业务的生产系统。该项目一方面用于展示，另一方面也用于记录我的学习过程。

# 当前功能

普通用户可以注册、登录、维护自己的资料，按地区、关键词和评分范围查询充电站，查看站点下的不同类型充电桩，然后开始和结束一次充电。

充电完成后，用户可以查看历史订单，并对使用过的站点进行评价。同一个用户对同一个站点只保留一条评价记录，再次提交时会更新原评价。

管理员端主要包括：

- 用户管理；
- 站点和充电桩管理；
- 全部充电订单查询；
- 结束异常状态的订单；
- 回复、隐藏和恢复用户评价；
- Dashboard统计数据；
- 导出数据为CSV文件；
- 管理员操作日志查询和导出。

项目界面主要面向中文用户，因此页面、状态文案和错误提示统一使用中文。

# 一些项目的实现

1、 有关于充电

开始充电时，需要同时确认以下情况：

- 用户处于可用状态；
- 用户现在没有其他进行中订单；
- 站点处于运营状态；
- 充电桩处于可用状态；
- 充电桩没有被其他订单占用。

这些检查放在事务中执行，并按照：

User
→ Location
→ Charging Station

的顺序加锁。

数据库层还使用两个生成列：

active_user_id
active_station_id

当订单状态为charging时，它们分别映射到用户和充电桩ID，再通过UNIQUE约束保证：

- 同一个用户最多只有一条进行中订单；
- 同一个充电桩最多只有一条进行中订单。

2、 结束充电用了不同的并发处理方式

开始充电需要协调多个对象，所以使用事务和行锁。

结束充电的核心竞争主要集中在一条订单记录，因此采用：

UPDATE ...
WHERE charge_record_id = ?
AND status = 'charging'

这样的条件更新。

如果两个请求重复结束同一订单，只有一个请求能够真正修改记录。

3、 订单保存费率快照

订单开始时会保存：

hourly_rate_snapshot

而不是在结束时重新读取充电桩的当前价格。

这样管理员后来调整费率时，只影响新的订单，不会改变已经开始或已经完成的订单结算结果。

4、 站点和充电桩状态不是简单开关

站点进入维护时，可用的充电桩会进入维护状态；站点停用时，可用和维护中的设备都会停用。

但站点重新恢复运营后，设备不会自动恢复为可用。

这是故意做成非对称的：自动停用属于安全收紧，但自动恢复可能把原本就因为硬件问题而维护的设备重新启用。

# 项目结构

项目没有使用Laravel等PHP Web框架。

目前的结构更接近：

public 页面入口
        ↓
Service
        ↓
Repository
        ↓
MariaDB

public下的PHP页面负责处理一次 HTTP 请求，包括参数读取、认证、CSRF、调用Service、Flash消息和跳转。

Service负责业务规则、事务和跨Repository协调。

Repository负责SQL、Prepared Statement、数据查询和对象映射。

Model保存业务对象的数据和一些只依赖自身状态的行为。

另外，app/Core中放置了一些基础功能。

这套结构不是完整的MVC框架，并没采用严格正式的架构。它只是我在这个项目规模下逐步形成的一套原生PHP分层方式。

# 本地运行

我开发时使用的环境是：

PHP 8.2
MariaDB / MySQL
Apache
XAMPP

1、 Clone 项目
git clone <repository-url>

把项目放到Web Server可以访问的目录中。

例如 XAMPP：

htdocs/easy_ev_charging

2、 创建数据库

执行：

database/easy_ev_charging_system_schema.sql

如需演示数据，再执行：

database/easy_ev_charging_system_demo_data.sql

3、 数据库配置

默认数据库配置位于：

config/database.php

项目优先读取以下环境变量：

DB_HOST
DB_PORT
DB_DATABASE
DB_USERNAME
DB_PASSWORD
DB_CHARSET

本地XAMPP环境也保留了开发用的默认配置。

4、 创建管理员

项目没有提供Web管理员注册页面。

可以在命令行运行：

php tools/create_admin_user.php

按照提示创建管理员账号。

# 测试

运行全部测试：

php tests/run.php

当前测试包括：

31 个 Unit Test
91 个 Feature Test
122 个测试

Feature Test使用独立的_test数据库，并在测试之间重建Schema，以减少测试数据互相影响。

这些测试覆盖了认证、用户管理、充电流程、站点和设备状态、评价、登录限流、查询以及管理员操作日志等部分。

需要说明的是，当前测试主要验证顺序业务流程，还没有实现真正的双数据库连接并发测试，因此不能因为 122 个测试通过就认为所有并发场景都已经被实际模拟。

# CLI工具
tools/create_admin_user.php

创建管理员账户。

tools/generate_password_hash.php

单独生成PHP Password Hash。

tools/clear_logs.php

清理运行日志或测试日志，也可以按时间保留最近的日志。

tools/generate_final_report.php

运行项目最终检查并生成Markdown报告。

最终检查除了运行测试外，还会检查PHP和JS语法、部分CSRF覆盖、数据库表、SQL直接调用位置、CSS Class使用候选以及一些关键安全和业务规则。

它只是一个针对这个项目写的检查工具，不是通用静态分析器或安全扫描器。

# 日志

运行日志位于：

storage/logs/

分为：

app.log
error.log
security.log

另外，管理员业务操作记录保存在数据库表：

admin_operation_logs

两者用途不同：

运行日志
——系统发生了什么

管理员审计
——哪个管理员做了什么

仓库中保留日志文件结构，但提交前会清空实际运行内容。

# 目前还没解决好的问题

这个项目还有很多可以继续改的地方。

目前我比较在意的几个问题是：

- 部分页面仍然自己重复组装Session、AuthGuard、Repository 和 Service，页面层结构还没有完全统一；
- 核心充电流程的并发保护比较完整，但部分低频后台编辑路径仍然存在检查后写入的理论竞争窗口；
- ChargeRecord保存了费率快照，但没有保存 Location 历史归属快照，设备迁移可能影响部分历史统计的解释；
- 管理员审计写入失败时，不同页面的处理方式还没有完全统一；
- 自动测试没有真正的双连接并发测试；
- 目前没有浏览器E2E测试和压力测试。
